## 為什麼需要多 Agent？

### 核心驅動力：並行 × 規模 × 分工

矽谷工程師的共識：多 Agent 不是為了「看起來更 AI」，而是解決三個具體問題。

| 驅動力 | 說明 |
|--------|------|
| ⚡ **並行速度** | 同一任務拆成 4 個子任務，單 Agent 串行 20 分鐘 → 多 Agent 平行 ~7 分鐘（含協調開銷） |
| 🔭 **突破 Context 上限** | 研究類任務資訊量超過單一 context window，各 Subagent 各自壓縮再回傳，等同分散式記憶體 |
| 🎯 **職責分離** | 不同 Agent 配備不同工具、不同系統提示，避免單一 Agent 因工具過多而混淆 |

### 關鍵數據

| 數據 | 來源 |
|------|------|
| **+90.2%** | Anthropic 多 Agent 研究系統 vs 單 Agent Claude Opus 4 |
| **+81%** | Google Research：可並行任務的效能提升 |
| **−70%** | Google Research：強制套用多 Agent 於串行任務的效能退化 |
| **15×** | 多 Agent 相較於一般對話的 token 用量倍數 |

### ⚠️ 何時「不該」用多 Agent

- ✕ 線性流程（每步依賴上一步結果）
- ✕ 簡單聊天機器人、早期原型
- ✕ 需要所有 Agent 共享同一 context 的任務
- ✕ 大多數程式碼任務（真正可並行的部分比研究少很多）

## 三大編排模式深度比較

同樣是「多 Agent」，這三種模式在結構、適用場景與工程成本上截然不同。

---

### 👶 Subagent（子代理）
> **Parent 派工 → Children 執行 → Parent 彙整**

| 技術特性 | 說明 |
|----------|------|
| 結構 | 父子階層（Parent → N 個 Subagent） |
| 並行度 | 高（子代理彼此獨立並行） |
| 專業分工 | 無（子代理通常是通用型） |
| 協調成本 | 低（只向 Parent 回報，不互相溝通） |
| 除錯難度 | 中等（需追蹤每個子任務輸出） |

**✅ 適合場景**
- 大量同質性工作（例：為 20 個 module 各自寫測試）
- 任務可精確拆分、邊界清晰
- 需要隔離 context 的批次處理

**❌ 避免使用於**
- 任務間有依賴關係
- 需要 Agent 間即時協調
- 任務定義模糊

> 💡 **類比**：臨時派遣工——老闆交辦、各做各的、結果交回。

---

### 👥 Agent Team（代理團隊）
> **角色分工 + 持久協作 + 對等溝通**

| 技術特性 | 說明 |
|----------|------|
| 結構 | Peer Network（Engineer / Designer / PM 平行協作） |
| 並行度 | 中（依角色職責有不同節奏） |
| 專業分工 | 高（每個 Agent 有明確角色、專屬工具） |
| 協調成本 | 高（需設計溝通協議） |
| 除錯難度 | 複雜（需追蹤跨 Agent 對話） |

**✅ 適合場景**
- 長期、跨職能的複雜專案
- 需要「寫作 + 審核」等內部檢核的流程
- 需要不同專業知識庫的任務（法律、財務、技術）

**❌ 避免使用於**
- 一次性任務
- 對協調開銷敏感的情境
- 團隊設計比任務本身更複雜時

> 💡 **類比**：常設團隊——工程師、設計師、PM 持續協作多個專案。

---

### 🔄 Dynamic Workflow（動態工作流）
> **單一 Agent 自適應 → 即時調整路徑**

| 技術特性 | 說明 |
|----------|------|
| 結構 | 單 Agent（自我規劃、自我調整） |
| 並行度 | 無（本質上是串行的） |
| 專業分工 | 無（通用型推理） |
| 協調成本 | 無（不需協調） |
| 除錯難度 | 最簡單（單一執行脈絡） |

**✅ 適合場景**
- 步驟相互依賴、難以預先拆解
- 任務範圍不確定（探索性調查）
- 偏好最小故障面

**❌ 避免使用於**
- 有大量真正可並行的子任務
- 任務規模超過單一 context window

> 💡 **類比**：資深工程師獨立攻關——自己看情況調整、不需要交辦給別人。

## 選擇決策樹

```
任務可拆成獨立子任務？
├── 否 → Dynamic Workflow ✅
└── 是 → 子任務需要不同專業角色/工具？
          ├── 是 → Agent Team ✅
          └── 否 → 任務是一次性、邊界清晰？
                    ├── 是 → Subagent ✅
                    └── 否 → Agent Team ✅
```

> 💡 **黃金法則**：從 Dynamic Workflow 開始，有具體理由才升級到 Subagent 或 Team。  
> 不是因為「更多 Agent 感覺更強大」。

## 矽谷工程師踩過的坑

以下為工程師在生產環境中實際遇到的失敗模式，每個都有配對的解法。

---

### 💸 Token 爆炸｜嚴重度：高

**問題**：多 Agent 協調每次通訊都是一次 LLM call。10 個 Agent 透過 coordinator 協作，可能產生 50–100 次 LLM call，協調開銷可讓成本增加 3–10 倍。

**解法**：針對路由/協調用輕量模型；複雜推理才用旗艦模型。可節省最多 35% token 成本。

---

### 🔥 錯誤傳播（Error Cascade）｜嚴重度：高

**問題**：研究階段的幻覺 → 分析階段的幻覺 → 最終報告的幻覺。錯誤在 Agent 鏈上滾雪球。

**解法**：在關鍵節點插入人工審核或自動驗證 Agent；不要讓錯誤靜默傳遞。

---

### 👯 重工浪費（Duplicate Work）｜嚴重度：中

**問題**：Anthropic 研究系統早期：指令太模糊，3 個 Subagent 同時調查相同的 2025 供應鏈議題，沒有有效分工。

**解法**：為每個 Subagent 提供明確的「任務邊界」與「禁止重複調查的範圍」。

---

### 🐛 Debug 地獄｜嚴重度：高

**問題**：沒有追蹤工具時，無法判斷是哪個 Agent 出錯、在哪個步驟失敗。

**解法**：生產環境必須搭配 Langfuse、LangSmith 等 tracing 工具；把 Agent 視為需要可觀測性的分散式系統。

---

### 📏 過度工程（Over-engineering）｜嚴重度：中

**問題**：大多數任務用一個設計良好的 Agent 就能解決。多 Agent 加了成本與風險，卻沒帶來相應收益。

**解法**：問自己：「哪些子任務真的可以並行？」如果答案是「沒有」，就不需要多 Agent。

---

### 🏛️ Coordinator 做太多｜嚴重度：中

**問題**：Orchestrator Agent 自己承攬了所有推理工作，Subagent 只是做搬運。這不是多 Agent，只是「貴的單 Agent」。

**解法**：Hub-and-Spoke 模式——Coordinator 負責協調，不負責執行。執行工作完全下放給 Worker Agent。

## 生產環境工程實踐

彙整自 Anthropic 工程部落格、250 人規模工程團隊案例、以及社群實戰。

---

### ✍️ Prompt Engineering

- **像 Agent 一樣思考**：在 Console 用完整 prompt 和工具模擬執行，觀察每一步，而不是猜測行為。
- **任務描述要具體**：每個 Subagent 需要明確的目標、輸出格式、可用工具邊界、禁止重複調查的範圍。
- **規模與查詢複雜度掛鉤**：在 prompt 中內嵌縮放規則，讓 Agent 自動判斷該開幾個 Subagent。

### 🏗️ 架構設計

- **Hub-and-Spoke 模式**：Coordinator 分離「協調」與「執行」職責，避免 Orchestrator 過載。
- **模型選型有策略**：路由/分類用輕量模型（如 Haiku），深度推理才用旗艦模型（如 Opus/Sonnet）。
- **3–7 個 Agent 最佳**：少於 3 個可能不需要多 Agent；超過 7 個協調複雜度通常超過收益（除非使用階層結構）。

### 📊 可觀測性

- **把多 Agent 系統當分散式系統對待**：必須有 tracing（LangSmith、Langfuse）才能 debug。
- **在關鍵節點留下 checkpoint**：讓人類在中間步驟可以介入、修正錯誤。
- **評估系統獨立建置**：對每個 Agent 分開做單元評估，不要只看最終輸出。

### 🏢 規模化心法（來自 250 人工程團隊的真實案例）

- **把 Agent 當新進工程師**：需要 onboarding 文件、編碼規範、架構慣例。
- **Agent-facing 文件優先於內部 wiki**：實際使用後，面向 Agent 的文件比 Confluence 維護得更好。
- **平台團隊專門維護技能庫（skills/rules）**：編碼內部架構慣例，讓所有 Agent 遵循。

## 現實狀況 & 未來方向

### ⚠️ 「協調」才是瓶頸，不是能力

CooperBench 研究顯示：GPT-5 和 Claude Sonnet 4.5 兩個 Agent 協作的成功率只有 **25%**，約為單 Agent 獨立完成的一半。問題不在智商，在溝通。

### 🏭 90% 企業部署 Agent，但只有 23% 成功規模化

技術門檻不是最大挑戰，**協調架構設計、可觀測性、成本控制**才是。

### 🔮 2026 年最值錢的技能：系統編排

不是 Prompt Engineering，而是設計、治理、整合人類 + AI Agent 混合工作流的能力。

### 📈 多 Agent 適合「高價值、重並行」任務

任務的商業價值必須覆蓋高達 15× 的 token 成本。**研究、法律分析、財務盡職調查**是目前最佳候選場景。

### 🤔 「多 Agent」定義還在演化

業界對「真正的多 Agent」沒有共識：
- **Anthropic 定義（寬鬆）**：多個 LLM 自主使用工具並協作
- **學術定義（嚴格）**：要求去中心化、自主性、局部視野

大多數生產系統其實是「委派式工作流」，不是真正的對等協作。
